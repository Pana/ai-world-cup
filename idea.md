# IDEA

2026 世界杯马上要开始了，往年都有各种预测世界杯冠军的营销活动（章鱼保罗，小狗等等）。今年其实可以做一个基于 AI 的预测活动，利用历史数据和当前球队状态来预测冠军。

重点是把各个 AI 模型都加入进来，让他们互相比赛，看看哪个模型的预测更准。如今大模型竞争激烈，各家公司的模型在不断更新，提升能力。这也是当今的热点话题。

## 预测积分规则

### 1. 统一计分口径

1. 所有比赛的比分预测均以 **常规时间 90 分钟（含补时）结束时的比分**为准。
2. 加时赛进球和点球大战比分不计入“准确比分”与“净胜球”。
3. 淘汰赛另外预测最终晋级或获胜球队，以及比赛是否进入加时赛、点球大战。
4. 各命中项可以累计得分。
5. 正式预测必须在比赛计划开球前锁定，锁定后不可修改。

### 2. 所有比赛的基础积分

| 预测项目 | 命中条件 | 得分 |
| -- | -- | --: |
| 胜负平 | 90 分钟的主胜、平局或客胜正确 | 2 分 |
| 净胜球 | 预测的主队进球数减客队进球数与实际相同 | 额外 +1 分 |
| 准确比分 | 90 分钟主客队比分完全正确 | 额外 +2 分 |

单场基础积分最高为 5 分。

示例：预测 `2:1`。

- 实际 `2:1`：胜负平 2 分 + 净胜球 1 分 + 准确比分 2 分，共 5 分。
- 实际 `3:2`：胜负平 2 分 + 净胜球 1 分，共 3 分。
- 实际 `1:0`：胜负平 2 分 + 净胜球 1 分，共 3 分。
- 实际 `2:0`：只命中胜负平，共 2 分。
- 实际 `1:1`：全部未命中，0 分。

### 3. 淘汰赛附加积分

| 预测项目 | 命中条件 | 得分 |
| -- | -- | --: |
| 最终晋级/获胜球队 | 正确预测该场最终晋级或获胜球队 | 额外 +2 分 |
| 进入加时赛 | 正确预测比赛在 90 分钟后进入加时赛 | 额外 +1 分 |
| 进入点球大战 | 正确预测比赛通过点球大战决胜 | 额外 +1 分 |

如果预测进入点球大战，也视为同时预测比赛进入加时赛；两项均命中时可累计获得 2 分。

淘汰赛单场最高为 9 分：基础积分 5 分 + 晋级球队 2 分 + 加时赛 1 分 + 点球大战 1 分。

示例：90 分钟实际 `1:1`，加时赛后 `2:2`，法国通过点球大战晋级。模型预测 `1:1`、法国晋级并进入点球大战，则获得：

- 胜负平 2 分
- 净胜球 1 分
- 准确比分 2 分
- 晋级球队 2 分
- 加时赛 1 分
- 点球大战 1 分
- 合计 9 分

### 4. 天眼预测：赛事前三名

在小组赛结束、32 强对阵确定后，各模型提交一次最终前三名预测。

| 预测项目 | 命中条件 | 得分 |
| -- | -- | --: |
| 冠军 | 球队及冠军位置完全正确 | 30 分 |
| 亚军 | 球队及亚军位置完全正确 | 20 分 |
| 季军 | 球队及季军位置完全正确 | 10 分 |
| 前三球队位置错误 | 预测球队进入前三，但具体位置错误 | 每队 5 分 |

同一支球队只取一个最高得分，不重复获得“位置正确”和“位置错误”分。天眼预测最高 60 分。

### 5. 异常比赛处理

1. 比赛取消：该场所有预测作废，不计分。
2. 比赛延期：使用新的开球时间，系统可以重新开放预测；旧预测版本必须保留。
3. 比赛中断后恢复：等待官方确认最终结果后计分。
4. 官方更正结果：重新计分，并保留旧计分记录和计算版本。
5. 模型调用失败或超过锁定时间：记为未提交，得 0 分，不允许赛后补交。

### 6. 排行榜同分规则

总积分相同时，依次比较：

1. 准确比分命中次数；
2. 净胜球命中次数；
3. 胜负平命中率；
4. 淘汰赛晋级球队命中次数；
5. 未提交场次更少者优先；
6. 仍然相同则并列。

每场比赛结束并完成官方结果确认后，重新计算各模型积分和总排行榜。

## 数据表

原始设想中的“球队、比赛、用户、大模型、预测、得分”六类表覆盖了核心对象，但不足以支持赛事阶段、小组关系、模型版本、Prompt 配置、预测调用追踪、可配置计分规则和用户竞猜。

数据库统一使用 **MySQL 8.0**。

AI 预测核心设计包含 15 张表：

1. 赛事届次 `tournaments`
2. 比赛阶段 `stages`
3. 小组 `groups`
4. 球队 `teams`
5. 赛事参赛球队 `tournament_teams`
6. 比赛 `matches`
7. AI 模型 `ai_models`
8. 模型配置 `model_configs`
9. Prompt 版本 `prompt_versions`
10. 比赛 Prompt 绑定 `match_prompt_versions`
11. 预测调用记录 `prediction_runs`
12. 单场比赛预测 `predictions`
13. 赛事前三名预测 `tournament_predictions`
14. 计分规则 `scoring_rules`
15. 预测得分明细 `prediction_scores`

当前 MVP 还实现了任务审计 `job_runs`、匿名用户 `user_profiles` 和用户竞猜
`user_match_predictions`。正式账号、跨设备身份和完整积分流水仍属于后续升级范围。

详细字段、约束、索引和表关系见：[database-design.md](database-design.md)。

## 系统运行核心步骤

1. 首先我会爬取比赛的排表。
2. 在每场比赛的头一天，通过 OpenRouter 统一调用各家大模型，对比赛进行预测。
3. 当比赛结束之后，对各个大模型的预测做一个打分。

整体，这就是一个核心的循环。那么针对每场比赛，只需要做这样的循环就行了。

## 大模型调用方式

1. 系统使用 **OpenRouter** 作为统一的大模型调用网关。
2. 每个参赛模型仍按原始厂商和具体模型版本独立记录、预测和计分。
3. OpenRouter API Key 通过环境变量 `OPENROUTER_API_KEY` 提供，不写入代码、数据库或 Git。
4. 每次预测调用必须保存 OpenRouter 模型标识、请求参数、原始响应、Token 用量、成本、耗时和错误信息。
5. 模型参数发生变化时创建新的模型配置版本，不能覆盖已有比赛的历史配置。
6. 正式环境应设置超时、有限次数重试和幂等键，防止重复生成正式预测。

## Prompt 版本机制

1. Prompt 独立于模型配置进行版本管理，例如 `match-prediction-v1`、`match-prediction-v2`。
2. 每个版本保存完整的 System Prompt、User Prompt 模板、JSON Schema、版本说明和内容哈希。
3. 已发布并用于正式比赛的 Prompt 版本不可修改；任何文字、字段或规则变化都必须创建新版本。
4. 每场比赛必须在开始调用模型前显式绑定一个 Prompt 版本。
5. 同一场比赛的所有参赛模型必须使用同一个 Prompt 版本和同一份数据快照。
6. Prompt 版本切换只影响尚未锁定的比赛。例如前四场绑定 V1，从第五场开始绑定 V2。
7. 已经使用 V1 完成预测的比赛不得因为启用 V2 而重新预测。
8. 每次 OpenRouter 调用记录必须保存实际使用的 Prompt 版本 ID、最终渲染后的 Prompt 和数据快照哈希。
9. 修改 Prompt 的原因和预期改进必须写入版本变更说明，便于后续比较不同版本效果。
10. Prompt 新版本只能用于未来尚未锁定的比赛，不能看到比赛结果后回头更换版本或重跑正式预测。
11. 上线新版本前，可以用历史数据或不计分的影子任务测试，但测试结果不能覆盖正式预测。
12. 排行榜和比赛详情页应显示 Prompt 版本，分析准确率时也应支持按版本分组。
